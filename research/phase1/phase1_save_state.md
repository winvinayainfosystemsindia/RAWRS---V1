# 1. Executive Overview

**RAWRS** (Remediation Automation Workflow & Review System) is a local-first document remediation pipeline built for **WinVinaya Foundation**, an organization that performs accessibility remediation of educational/academic PDFs into accessible DOCX format for visually-impaired and other readers.

**Why it exists:** WinVinaya's human remediators currently take a PDF and manually rebuild it in Word — applying heading styles, alt text, page markers, footnote linking, table reconstruction, etc. — following two checklists that govern that manual process (`docs/Checklist for Document Remediation1.docx`, `docs/ChecklistBeforeSubmittingDoc.xlsx`; see §9). This is slow and repetitive. RAWRS automates the deterministic, rule-based parts of that workflow (structure detection, heading hierarchy, page markers, footnote detection, image extraction/filtering, alt-text placeholder infrastructure) so the human remediator's effort concentrates on the parts that genuinely need judgment (descriptive alt text, table reconstruction, equation handling, final accessibility sign-off).

**Phase 1 objective** (`docs/PHASE1_SCOPE.md`):
```
PDF → Structured Markdown → Accessible DOCX
```
North star metric: **Human Minutes Per 100 Pages** — every Phase 1 decision should reduce this while maintaining accessibility quality.

**Current maturity level:** Phase 1's automated pipeline is functionally complete and well-tested (**588 tests collected** as of direct re-run on 2026-06-24, up from the `567` figure recorded in this file's prior synthesis pass — the +21 are `tests/test_footnote_detector.py::TestBug005SpanSuperscriptDetection`/`TestBug005RealRegressionPdf`, added 2026-06-23 alongside the span-level footnote fix; see §6/§10 for why this also differs from the `486` figure still recorded in `docs/PHASE_STATUS.md`, which is now doubly stale). A FastAPI backend and a Next.js frontend now exist (§12) — this is a **major undocumented-in-`docs/` reality**: `docs/CURRENT_STATE.md` and `docs/ARCHITECTURE_CURRENT.md` both currently say "no frontend, no backend/API, `src/main.py` is empty." That was true when those files were last reconciled; it is no longer true. See §8/§10 for the documentation-staleness implications.

Three real architectural gaps were found via deep regression auditing in an earlier session (paragraph reconstruction, heading detection, footnote detection). **Two of the three were fixed and verified in that same session (paragraph reconstruction → bug_001, heading detection → bug_002); the third (footnote/endnote span-level information loss) was root-caused and design-reviewed but explicitly left unimplemented pending authorization.** That authorization was given and the fix was implemented and verified on 2026-06-23 (`bug_005`/`feature_005_span_level_text_model` — see §7/§11 for the full record). # 2. Current Architecture

## Documented vs. actual pipeline order

`docs/ARCHITECTURE.md` (canonical/intended) states:
```
PDF → Parser → OCR → Structure Detection → Footnote/Endnote Detection → Heading Detection → Image Extraction → Markdown Generation → Validation → DOCX Generation → Output
```

The **actual** order, per `src/pipeline/phase1_pipeline.py` (confirmed by direct reading, not by doc trust) and recorded as fact in `docs/ARCHITECTURE_CURRENT.md`:

```
1. Parse PDF                                              src/parser/pdf_parser.py
2. Extract Text (direct, born-digital)                    src/ocr/extractor.py
   2a. Route pages DIRECT_TEXT vs OCR_REQUIRED             src/ocr/router.py
   2b. Docling OCR on OCR_REQUIRED pages                   src/ocr/docling_engine.py
   2c. Surya OCR fallback on pages Docling left empty      src/ocr/surya_engine.py
3. Detect Structure                                        src/structure/structure_detector.py
   3a. Detect Footnotes/Endnotes (reuses Structure's blocks) src/footnotes/footnote_detector.py
4. Extract Images                                          src/images/image_extractor.py
5. Detect Headings                                         src/headings/heading_detector.py
6. Generate Markdown                                       src/markdown/markdown_builder.py
7. Generate DOCX                                           src/docx/docx_generator.py
8. Run Validation                                          src/validation/validator.py
```

**Two deliberate, tracked deviations from the canonical diagram** (recorded in `DECISIONS_LOG.md`, never picked up for realignment despite nothing technically blocking it):
1. Image Extraction runs **before** Heading Detection (canonical has Heading Detection first).
2. Validation runs **after** DOCX Generation (canonical has Validation before DOCX Generation) — meaning the validation report a human reviewer reads is generated *after* the DOCX artifact already exists, not as a gate before it's produced.

## Per-stage detail

### Parser — `src/parser/pdf_parser.py`
- **Purpose:** Load the PDF, extract per-page metadata, build the initial `Document`/`Page` model tree.
- **Input:** file path.
- **Output:** `Document` with `Page` objects (no text yet).
- **Known limitations:** none specific; this stage is structurally simple.

### OCR — `src/ocr/extractor.py`, `router.py`, `docling_engine.py`, `docling_config.py`, `surya_engine.py`, `surya_config.py`
- **Purpose:** Get usable text onto every page, preferring the cheapest/most-certain method.
- **Order:** Direct text extraction (PyMuPDF, zero recognition uncertainty, `OCRConfidence.HIGH`) → per-page routing (`DIRECT_TEXT` vs `OCR_REQUIRED`, based on extracted-text length <20 chars or >10% control/replacement-char ratio — **not** PDF metadata) → Docling (`force_full_page_ocr=True`, `OCRConfidence.MEDIUM`) on `OCR_REQUIRED` pages → Surya fallback (`OCRConfidence.LOW`) only on pages Docling left empty.
- **Input:** `Document` with empty pages.
- **Output:** `Document` with `Page.cleaned_text`/`raw_text` populated, `OCRConfidence` set per page.
- **Known limitations:**
  - `force_full_page_ocr=True` is required because Docling's *default* layout-driven OCR mode returned **zero text** on real benchmark pages confirmed (by direct PDF inspection) to contain genuine prose — this is itself a discovery, not a default choice, recorded in `DECISIONS_LOG.md`.
  - **Surya's real runtime dependency is a major, previously-mis-documented discovery:** `surya-ocr==0.20.0` ("Surya2", a VLM rewrite, not classical Surya) auto-selects `vllm` (GPU) or `llamacpp` (CPU) as its inference backend via `SuryaInferenceManager`. On this CPU-only deployment, that means Surya OCR actually works by **spawning the real upstream `llama-server` binary** and serving the `surya-2.gguf` vision-language model through it over a local OpenAI-compatible HTTP API — confirmed with live evidence (a `LLAMA_CPP_BINARY` env var pointing at a real `llama-server.exe`, a cached `surya-2.gguf`+`surya-2-mmproj.gguf` pair from Hugging Face Hub, and a `llama-server` runtime log showing real per-token generation timing). An **earlier audit pass got this wrong** — it found no `llama-cpp-python` entry in `requirements.txt` and no llama.cpp string in RAWRS's own `src/`, and incorrectly concluded Surya doesn't use llama.cpp at all. That incorrect "correction" was itself written into `OCR_RULES.md`, `TECH_STACK.md`, `PHASE_STATUS.md`, and `DECISIONS_LOG.md`, and has since been reversed in all four. **Lesson recorded for future audits:** verifying "no reference in RAWRS's own code" is not equivalent to "no dependency exists" — the dependency lived one layer down, inside the third-party package's own runtime backend selection. `requirements.txt` now pins `surya-ocr==0.20.0` specifically so this doesn't silently drift on a future clean install.

### Structure Detection — `src/structure/structure_detector.py`, `src/structure/layout_signals.py`
- **Purpose:** Persist per-line (not per-paragraph) layout metadata — bbox, font size, bold flag, PyMuPDF block index, emission order — into `Document.blocks: List[TextBlock]`, as a foundation for later phases (footnote detection, reading-order validation, paragraph reconstruction) so they don't each independently re-derive it.
- **Input:** `Document` with extracted text.
- **Output:** `Document.blocks` populated.
- **Design note (load-bearing for §11):** `TextBlock` is **deliberately line-granularity**, with exactly one scalar `font_size: Optional[float]` and one scalar `is_bold: Optional[bool]` per line (max-of-line size; >50%-char-majority bold vote). This was a deliberate Phase H choice that served column-clustering and paragraph-reconstruction needs well — but is the direct, confirmed root cause of the footnote-detection gap in §7/§11.
- **Known limitations:** purely additive — "never reads or alters reading order, columns, tables, or equations" (its own docstring). No multi-column detection, no table detection.

### Footnote/Endnote Detection — `src/footnotes/footnote_detector.py`
- **Purpose:** Detect inline footnote/endnote markers and their note bodies, link marker→body by number (per-page scoping for footnotes, document-wide for endnotes via a "Notes"/"Endnotes" section heading), populate `Document.footnotes`.
- **Input:** `Document.blocks` (plus one independent PDF re-open, for page height only — to test "is this candidate body in the bottom quarter of the page").
- **Output:** `Document.footnotes: List[Footnote]`.
- **Two marker-detection signals as of `bug_005` (fixed 2026-06-23 — see §7/§11 for full history):**
  1. **Literal Unicode superscript glyph** (U+00B9/U+00B2/U+00B3/U+2070/U+2074–U+2079) glued onto a word — the original, pre-existing path.
  2. **Span-based signal** (new): `_find_span_marker_candidates()` fires when a span is 1–3 plain digits, carries PyMuPDF's `TEXT_FONT_SUPERSCRIPT` flag bit, is strictly smaller than the largest span on its line, and is glued onto the immediately preceding span with no space — the real-world-common encoding the literal-glyph path always missed. Both paths feed the same `detect_footnotes()`, so endnotes are fixed by the identical change with no separate code.
  - Each detected marker now also carries `Footnote.anchor_offset: Optional[int]` (exact character position within its source line), computed by `_span_marker_offset()` with a fail-closed safety check (returns no candidate rather than risk a wrong position). This was added **not** because the design review anticipated it, but because verification surfaced a second, independent bug — see §7 — in how markers get substituted into Markdown.
  - **Still a known limitation:** `Span` (and therefore signal 2) only ever populates for `DIRECT_TEXT` pages; whether Docling/Surya OCR output exposes an equivalent span-level signal remains a separate, unscoped, unverified question (carried over from the original `feature_005` design review, §11).

### Heading Detection — `src/headings/heading_detector.py`
- **Purpose:** Detect H1–H6 hierarchy using a layered rule chain (numbering patterns → positional H1 slot → keyword/chapter patterns → bold-relative-to-body layout signal → **bug_002's new fallback tier**, see §6/§7), and generate the mandatory H6 page marker for every page. **`feature_007` (2026-06-25):** after classification, a bold anchor line absorbs up to 4 following same-layout lines that are confirmed continuations (same block, or cross-block within a calibrated `gap_ratio` window of -0.20 to +0.45), repairing multi-line-wrapped headings that were previously split into a correct heading plus spurious extra heading fragments.
- **Input:** `Page.cleaned_text`/`raw_text`, plus two independent re-opens of the source PDF for layout signal (`_build_layout_index`, now also carrying block index per line) and the new fallback-tier signal (`_build_fallback_tier_index`).
- **Output:** `Document.headings: List[Heading]`, in document-wide reading order.
- **Known limitations:** font *color* is never inspected as a detection signal (only applied unconditionally on output); the "Article" journal-kicker false-H1 (bug_003, open, not yet fixed — see §6); no multi-tier bold-size differentiation beyond H1/H2 fallback level; `feature_007`'s cross-block fallback window is calibrated against only 1 positive example corpus-wide (Aims of Education's title) — revisit if a future document misbehaves.

### Image Extraction — `src/images/image_extractor.py`
- **Purpose:** Extract images, filter out non-content (background/full-page ≥85% page area, extreme-aspect slivers, tiny rasters <16px, byte-digest duplicates, non-referenced images), link nearby figure captions (regex + ≤36pt proximity match, one block claimed per image), generate a deterministic placeholder alt-text string, persist bbox.
- **Output:** `Document.images: List[Image]` (each optionally carrying a composed `Image.figure: Optional[Figure]`).
- **Known limitations — confirmed this session, not previously tracked:** `python-docx`'s `add_picture()` raises `UnrecognizedImageError` on **CMYK JPEGs** (Adobe APP14 marker) even though PIL opens them fine. On the Brinkman regression PDF, 3 of 4 successfully-extracted images silently fail to embed into the final DOCX — `inline_shapes` count is 1, not 4. The validation report's `IMAGE_004` (alt-text-pending) still fires for all 4, which is misleading: it implies 4 images are present in the DOCX when only 1 is. Neither `IMAGE_001` ("file is missing") nor any other rule catches this, since the file genuinely exists on disk — it just fails to embed. **Not yet filed as a numbered bug** — see §6.

### Markdown Generation — `src/markdown/markdown_builder.py`
- **Purpose:** Build canonical Markdown — the project's stated "source of truth for downstream processing." Page markers, page breaks, heading substitution, footnote syntax, image+alt-text embedding, and (since bug_001's fix) **paragraph reconstruction** via `src/structure/paragraph_grouper.py::group_into_paragraphs()`.
- **Known limitations:** raw Unicode ligature glyphs (ﬀ/ﬁ/ﬂ/ﬃ) are preserved verbatim rather than decomposed to ASCII pairs — confirmed 95 occurrences in the Brinkman regression PDF's output, not yet fixed (candidate bug_004, open).

### Validation — `src/validation/validator.py`
- **Purpose:** "AI Proposes, Validation Decides" — the project's core safety principle (`docs/VALIDATION_RULES.md`). Read-only; never modifies content.
- **16 implemented rule IDs** (current as of this session — `docs/PHASE_STATUS.md`'s table is stale, missing `DOC_004`, `OCR_001`, `OCR_002`, see §10): `DOC_001`–`DOC_004`, `HEADING_001`–`HEADING_004`, `PAGE_001`–`PAGE_003`, `IMAGE_001`–`IMAGE_004`, `OCR_001`, `OCR_002`, `NOTE_001`, `NOTE_002`.
- **Known gaps:** Figure Validation (missing captions, unlinked references, missing numbering) has no rule ID at all. Broken-word detection has no rule ID.

### DOCX Generation — `src/docx/docx_generator.py`
- **Purpose:** Convert Markdown to an accessible DOCX — real Word `Heading 1`–`Heading 6` paragraph styles (Navigation Pane support confirmed working), page markers/breaks, centered inline images with `docPr` alt-text attributes, footnote/endnote preservation via `w:hyperlink`→`w:bookmark` (**not** native `w:footnote`/`w:endnote` OOXML elements — a real, working, clickable mechanism, just not the one Word's auto-numbered footnote pane reads), `_safe_run_text()` as a last-resort XML-sanitization guard (Layer 3, see §7).
- **Known limitations:** no native Word footnotes; CMYK image embedding gap (above); no table generation at all (Markdown table-like content renders as flattened cell-text paragraphs, since no table detection exists upstream either).

# 7. Major Discoveries

### XML Sanitization Architecture C

**Original failure:** A production PDF crashed `generate_docx()` with `ValueError: All strings must be XML compatible` — raised inside `lxml`'s compiled core the instant an OOXML node is set to a character XML 1.0 disallows (e.g. U+0002, from a broken PDF font/ToUnicode mapping — "p ≤ 0.001" extracted as "p\x020.001").

**Root cause:** tracing every text path found **three independent PyMuPDF read passes** (direct extraction, structure detection's dict-mode read, heading detection's *own separate* dict-mode read) — only the first had *any* cleanup (`normalize_whitespace()`, whitespace-only, never XML-legality). Figure captions and footnote/endnote text both read from the second pass and would have stayed vulnerable to a source-only fix; future AI-generated text (alt text, equations, tables) would be unprotected *by construction*, since generated text has no PDF-extraction call to attach a sanitizer to.

**Three-layer defense (Architecture C, chosen over source-only and export-boundary-only):**
1. **Layer 1** (`src/utils/text_sanitization.py`) — sanitizes at every point text first enters the model (extractor, docling_engine, surya_engine, structure_detector).
2. **Layer 2** (`validator.py`, rule `DOC_004`, severity **WARNING not ERROR** — by the time it can fire, Layer 1 has already removed the character and the document has already generated successfully, so "processing quality is compromised" is false by construction) — discloses every place Layer 1 had to act via `Document.sanitization_events`.
3. **Layer 3** (`docx_generator.py`, `_safe_run_text()`) — last-resort guard at every OOXML-text call site; logs loudly if it ever actually changes something, since that signals a real upstream gap.

**Final status:** resolved and implemented; live-API-confirmed end-to-end (started the real `uvicorn` server, POSTed a synthetic `\x01`/`\x02` repro PDF through `/api/documents`, downloaded the resulting DOCX — zero control characters — and confirmed `DOC_004` correctly named the removed codepoints with page attribution).

### Paragraph Reconstruction

**Bug 1 (extraction-level):** PyMuPDF mis-segments one justified PDF line into 8 fragments when inter-word gaps are absolute-positioning jumps rather than literal space glyphs — confirmed via raw bbox dump (identical y-coordinates across all 8 fragments, no bridging space-character span at any boundary). Upstream of RAWRS; inherited verbatim by both of RAWRS's independent re-derivation paths.

**Bug 2 (rendering-level, the pervasive one):** `markdown_builder.py`'s `_render_page_body()` had no paragraph-joining logic at all — every line became its own markdown block.

**Final design (Option B over A/C, see §6 table):** geometry-grounded grouping on `TextBlock.source_block_index` + bbox x-continuity, with a vertical-gap fallback reusing the validator's already-tested 1.5×-median-line-height threshold — chosen because it fixes both bugs with one mechanism grounded in already-extracted data, is the same prerequisite signal future column/table/equation work would need anyway, and avoids the "naive global line joining" lexical-heuristic risk (Option A) without committing to a full new pipeline stage that wasn't scheduled (Option C).

### Heading Detection (bug_002)

**Brinkman findings:** 12 real section headings used a distinct embedded font subset (`AdvP7D0F`, 12pt) with no "bold" substring and no PyMuPDF bold flag — defeating the existing bold-gate, which checks `"bold" in font_name.lower() or flags & 16`.

**Font-family discovery:** confirmed via direct `page.get_text("dict")` span dump — `AdvP7D0F` (heading) vs. `AdvTimes` (body), neither bold-flagged by either signal.

**Fallback tier, exactly as implemented:** font≠body AND (font,size) recurs ≥2× AND not-H1-slot AND length-guard (reused from the existing global check) AND alphabetic AND **sole-line-block** AND **size≥body-size**.

**Sole-line-block rule, why it's safe:** real headings are reliably emitted as 1-line PyMuPDF blocks; a running header sharing a block with its page number (e.g. "Brinkmann"+"343" on the identical baseline, confirmed via direct dump — same mechanism as paragraph-reconstruction Bug 1) is not. Verified this restriction is **only** enforced inside the new tier, never globally: "Chapter 9"/"Chapter 7" are real headings that are **not** sole-line (they share a 3-line masthead block with the chapter title), yet remain correctly detected because the H1-positional-slot rule resolves them *before* the new tier is ever reached — confirmed empirically against the real Calderhead/Fullan&Hargreaves PDFs, not just argued structurally.

**Size guard, why it was added (a real regression, not speculative):** verification found table/figure captions and table-footnote lines (8–9pt, distinct recurring font, sole-line) satisfied all 6 originally-audited gates exactly like real headings — Brinkman's 12 real headings are 12pt against a 10pt body; every false positive found was 8–9pt, smaller than body. Added as a 7th condition, disclosed to the user as a deviation from the literal spec before implementing (per the established checklist-impact-analysis discipline, §9).

**A second real regression, also found and fixed during implementation:** "James Calderhead" and "Michael Fullan and Andy Hargreaves" (bylines) initially got wrongly promoted, because "Chapter 9"/"Chapter 7" (non-sole, 3-line masthead block) shared their exact 14pt-Helvetica signature and inflated the recurrence count past threshold for the unrelated, separately-blocked, sole-line byline beneath them. Fixed by counting recurrence only from sole-line contributions.

### Footnote Audit

**Why the detector is not the problem:** `footnote_detector.py`'s logic is internally consistent and correct against the only signal it's given. Its own docstring already states the limitation honestly: detecting a plain-digit marker apart from incidental body-text digits "would need sub-line span data Phase H's `TextBlock` (line-granularity) doesn't persist."

**Span-level information loss, confirmed not assumed:** direct inspection of the real Brinkman footnote marker span: `text='1', size=7.04, flags=5 (SUPERSCRIPT|SERIFED), origin_y=329.39` vs. the adjacent body word `text='In-service', size=9.96, flags=4, origin_y=333.81`. PyMuPDF extracts the full distinguishing signal (smaller size, superscript flag, raised baseline). `TextBlock` never sees it: `font_size` is the **max** span size on the line (a single small superscript char can't move it even before any vote), `is_bold` is a >50%-char-majority vote (one character among ~40 can never flip it). The information is captured by PyMuPDF and destroyed by RAWRS's own aggregation, not missing from the source.

**`TextBlock` limitations, precisely:** one `font_size`, one `is_bold`, **no font name field at all**, no per-character/per-span data, no baseline-position data. A second, broader piece of evidence: the corpus's only math-like content (`r = .66, p < .001`) is encoded as 5 spans across 4 different fonts (`AdvTimes`, `AdvTimes-i` italic for variables, `AdvMT_SY` symbol font, `AdvMT_MI` math-italic) — all collapsed into one scalar pair. The loss isn't limited to sub/superscript; it's any span-level distinction within a line.

**Future implications (the reason `feature_005_span_level_text_model` exists):** PyMuPDF's flag set has **no dedicated subscript bit** (`TEXT_FONT_SUPERSCRIPT=1, ITALIC=2, SERIFED=4, MONOSPACED=8, BOLD=16` — confirmed exhaustively), so subscript detection would need pure baseline-position inference even after a fix, with a structurally weaker raw signal than superscript has. Equations inherit the identical root cause *plus* font-family loss and likely multi-baseline layout RAWRS doesn't model at all. Chemistry notation inherits it for its sub/superscript components. Citations are **conditional on style** — Brinkman's own parenthetical citations (`(Schweisfurth, 2013)`) are plain inline text, completely unaffected; Vancouver/Nature-style numbered superscript citations would hit the identical mechanism as a footnote marker. See §11 for the full standing initiative this spawned, and the subsection immediately below for its implementation.

### Footnote/Endnote Span Fix — Implementation (`bug_005`, 2026-06-23)

**What changed:** exactly the design review's Option A and 4-step migration (§11), carried through to working code in one session, on the real Brinkman regression PDF (`samples/regressions/bug_005_footnote_endnote_information_loss/`). Phase 1 (additive `Span` model + empty-default `TextBlock.spans`) was verified against the full test suite *before* any consumer was touched, confirming zero behavior change from the additive field alone. Phases 3–5 then gave `footnote_detector.py` a second marker-detection signal (`_find_span_marker_candidates()`) alongside the pre-existing literal-Unicode-glyph path, fixing footnotes and endnotes together since both share `detect_footnotes()`.

**A second bug, found only during verification, not predicted by the design review:** once real markers became plain digits instead of rare Unicode glyphs, `markdown_builder.py`'s old positionally-blind `line.replace(marker, ..., 1)` substitution started silently corrupting *unrelated* text — a different page's marker "1" matching the literal character inside "13" or inside the year "2010", several paragraphs away from the real marker. This is the kind of failure mode that specifically would not show up by inspecting the new detection logic in isolation; it only surfaced by regenerating the actual Markdown output and diffing it against expected prose. Fixed by adding `Footnote.anchor_offset` (the marker's exact character position, computed with a fail-closed safety check) and replacing all three substitution call sites with one shared `_substitute_markers()` — position-based where `anchor_offset` is valid, falling back to a blind replace bounded to *only that note's own anchor line* (never the whole paragraph) when it isn't, with replacements applied in descending-position order so one substitution's length change can't invalidate another's still-pending offset.

**Verification:** all 3 real markers on the Brinkman PDF detected via the new span signal and land on their exact real source position in the regenerated Markdown; the previously-corrupted unrelated numbers confirmed untouched by direct re-inspection. 21 new tests (`tests/test_footnote_detector.py::TestBug005SpanSuperscriptDetection`, `::TestBug005RealRegressionPdf`) cover the positive cases, a no-false-positive sweep (table-cell digits, running-header page numbers, the statistical notation `r = .66, p < .001` — none of which satisfy all four required span conditions simultaneously), and the end-to-end real-PDF case. Full suite: 588 tests collected (up from 567), all passing.

**What this does *not* close:** subscripts, equations, and chemistry notation are explicitly out of scope for this fix — they inherit the same root cause but need additional work the design review already flagged as unsolved (no PyMuPDF subscript flag bit; multi-baseline layout for equations). See §11's status update for what remains.

### Front-Matter Semantic Extraction (`bug_006` / `feature_006_front_matter_extraction`, 2026-06-24)

**What was found:** a "Scholarly Article Semantics Audit" on the Brinkman benchmark PDF found that the document's title, author byline, and institutional affiliation had no detection treatment at all — not a heading (wrong font-size-rank/keyword signal), not metadata, nothing. All three rendered as one undifferentiated run-on body paragraph in both the generated Markdown and DOCX.

**What changed:** a new, additive module, `src/frontmatter/front_matter_extractor.py`, deliberately isolated from `heading_detector.py` (no shared constants, no calls into it, no change to any of its classification tiers). It reads only `Document.blocks` (already persisted by Structure Detection, Phase H) — no new PyMuPDF re-open. Algorithm: find a "masthead-zone boundary" within page 1's first 20 lines (the first line matching `abstract`/`keywords`/`introduction`/`summary`); within that zone, optionally skip one short kicker line (e.g. Brinkman's "Article"); the title is the contiguous run of lines ≥1.3× the document's dominant body font size; the author is the contiguous run immediately after, strictly between body and title size, capped at 5 lines; the affiliation is everything remaining in the zone. Any step finding nothing (no boundary, no title) fails closed — the whole `FrontMatter` stays empty, which is the correct, expected outcome for a PDF with no title page (3 of the 4 benchmark PDFs).

Wired into `phase1_pipeline.py` Stage 3 (immediately after footnote detection), consumed by `markdown_builder.py` (renders a bold title / italic byline / plain affiliation block right after page 1's H6 marker, and suppresses those exact source lines from ordinary body rendering using the same exact-line-matching technique already used for footnote bodies and figure captions) and `docx_generator.py` (styled title/byline/affiliation paragraphs in the DOCX). A related one-line fix landed alongside it: `"keywords"` was added to `heading_detector.py::_H2_KEYWORDS`, since a PDF's literal "Keywords" line was previously falling through undetected as a heading — found during the same audit, same symptom class.

**Verification:** `tests/test_front_matter_extractor.py`, 18 tests — synthetic fixtures covering the full masthead zone, no-kicker, fail-closed-on-missing-boundary/missing-title, and author/affiliation split variants, plus `TestRealBrinkmanPdf`, an end-to-end test against the real Brinkman regression PDF confirming title/author both extract correctly from real PDF geometry.

**Process gap, corrected in this pass (2026-06-25):** this work was implemented and verified the same day it was found (2026-06-24), but never given a ticket number, never added to this file, `TASKS.md`, `PHASE_STATUS.md`, or `DECISIONS_LOG.md`, and no memory was saved — a deviation from this project's own standing convention (every other fix in this section has a `bug_NNN`/`feature_NNN` ticket). Retroactively assigned `bug_006`/`feature_006_front_matter_extraction` in this pass, mirroring the existing `bug_005`/`feature_005_span_level_text_model` symptom-ticket/architecture-ticket pairing. No code was changed in this pass — documentation and memory only.

# 11. Span-Level Architecture Discovery

## Why the footnote audit changed the roadmap

The footnote/endnote audit (§7) was scoped narrowly — "why does footnote detection fail on this one PDF" — and surfaced something much larger: **the entire pipeline downstream of `structure_detector.py` is working with less information than PyMuPDF actually provides**, by a specific, traceable, deliberate architectural choice (`TextBlock`'s line-granularity), not by an extraction gap.

```
PyMuPDF page.get_text("dict")
  → span-level data EXISTS: per-character-range font name, size, bold/italic/superscript flags, baseline origin
  → structure_detector.py / layout_signals.py::line_layout()
      → collapses to ONE (max-size, majority-bold) pair per LINE
      → RAWRS DISCARDS the rest, permanently, before any downstream module runs
  → TextBlock (Document.blocks) — what every later phase actually sees
```

This is not a future risk — it's the **confirmed, currently-active root cause** of a real detection failure (0 of 3 real footnotes detected on the Brinkman PDF), with direct span-dump evidence, not inference.

## Implications, per category (full audit detail in §7)

| Category | Affected by the same root cause? |
|---|---|
| Footnotes | **Yes — confirmed, this is the active bug.** |
| Endnotes | **Yes**, same marker-detection mechanism as footnotes. |
| Superscripts (general) | **Yes, identical mechanism** — any superscript (exponents, ordinals, markers) relies on the exact signal discarded at the same point. |
| Subscripts | **Yes, and structurally weaker** — PyMuPDF has no dedicated subscript flag bit at all; would need pure baseline-position inference even after a `TextBlock` fix. |
| Equations | **Yes, plus additional problems** — confirmed via the corpus's one real example (`r = .66, p < .001`, 5 spans across 4 fonts collapsed to one scalar pair); likely also affected by multi-baseline layout (fractions, radicals) PyMuPDF's line/span model may not even represent coherently. |
| Chemistry notation | **Yes, for its sub/superscript components** (H₂O, Na⁺, ¹⁴C), plus a related but separate symbol/glyph-mapping concern for reaction arrows etc. |
| Citations | **Conditional on source-document style**, not a fixed yes/no — parenthetical citation styles are completely unaffected (plain inline text); numbered-superscript citation styles (Vancouver/Nature) hit the identical mechanism as a footnote marker. |

## Status: design review COMPLETE, Steps 1–3/5 of the migration IMPLEMENTED 2026-06-23 (Step 4 still open)

Tracked as **`feature_005_span_level_text_model`**, blocking ticket **`bug_005`** (now closed for footnotes/endnotes/superscripts — see §6/§7). The full design-only review (trace the exact discard point, inventory every `TextBlock` consumer, design a minimal `Span` model, decide A/embedded vs. B/parallel vs. C/separate-artifact placement, evaluate impact on footnotes/endnotes/superscripts/subscripts/equations/chemistry/citations and on memory/performance/OCR/Markdown/DOCX, propose migration paths, determine multi-column/table independence) was completed first, preserved in full in memory (`project_span_level_design_review.md`). **Implementation was then authorized and carried out the same day (2026-06-23)** — this is no longer design-only.

**Conclusion: Option A**, as recommended and as built — `spans: List[Span]` embedded directly in `TextBlock` as an additive field (default empty list), not a parallel `Document.spans` list or a new pipeline stage. `src/models/span.py` now exists with exactly the fields proposed (`text`, `font_name`, `font_size`, `font_flags`, `baseline_y`, `bbox`).

**Two real findings from the original consumer inventory, confirmed still accurate post-implementation:** `TextBlock.is_bold` still has zero production consumers beyond its own test. `TextBlock.font_size` still has exactly one production call site (`footnote_detector.py`'s size-drop heuristic) — both migrated through cleanly with zero changes needed to either.

**One real gap the review surfaced, still unresolved post-implementation:** `Span` as built only ever populates for `DIRECT_TEXT` pages; whether Docling/Surya OCR output exposes an equivalent span-level signal remains a separate, unscoped investigation — not addressed by the `bug_005` fix, which only needed to work on the Brinkman PDF's direct-text pages.

**Multi-column/table independence: confirmed yes**, unaffected by the implementation — neither blocks the other.

**4-step migration, actual outcome vs. the original proposal:**
1. ✅ **Done.** Additive `Span` model + empty-default `TextBlock.spans` field, verified zero behavior change against the full suite before any consumer was touched.
2. ✅ **Done.** `spans` populated in `structure_detector.py`'s existing per-span loop (`_extract_spans()`), no new PDF pass.
3. ✅ **Done, plus an unplanned extension.** `footnote_detector.py::_find_span_marker_candidates()` added as the second marker-detection path. The original plan stopped here; verification then surfaced a second bug (markdown_builder.py's positional-replace corruption, §7) that required adding `Footnote.anchor_offset` and rewriting `_substitute_markers()` — not anticipated by the design review, found only by regenerating real output and diffing against expected prose.
4. ❌ **Not done, still open.** Re-evaluating `heading_detector.py`'s bug_002 fallback tier against a direct `Span.font_name` read was explicitly deferred as "its own deliberate follow-up" in the original plan, and nothing since has picked it up. This is now the natural next step for whoever wants to extend the span migration further (§14).

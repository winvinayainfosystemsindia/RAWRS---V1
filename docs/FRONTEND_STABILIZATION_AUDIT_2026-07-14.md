  # RAWRS Frontend Stabilization Audit — 2026-07-14 (Phase F-5.0)

Audit + consolidation only, per the ticket — no code changed, no refactoring, no visual redesign. This audit treats every completed frontend milestone (F-1 Completion Audit, F-2.1/F-2.2 Accessibility, F-3.1/F-3.2 Keyboard, F-4.1 Search Audit, F-4.2 Layout Audit, F-4.3 Persistence/Layout Quality, F-4.4 Mode Design Audit, F-4.5 Coverage Parity) as one system rather than nine separate deliveries. Grounded in direct inspection this session (`WorkspaceShell.tsx`, `DocumentWorkspace.tsx`, `ContextInspectorRail.tsx`, `PdfViewer.tsx`, `PdfViewportContext.tsx`, `FootnoteTable.tsx`/`ListPanel.tsx`/`CalloutPanel.tsx`, `useArrowKeyTabs.ts`, `ThemeProvider.tsx`, `app/` route structure) plus a dedicated independent verification pass (dead code, duplicate abstractions, error handling, keyboard consistency, theme-token consistency, loading/empty-state patterns, route/bundle sanity) covering every file this session had not yet directly read.

---

## 1. Overall Frontend Health Assessment

**The frontend is internally consistent and healthy.** No Critical findings. Exactly one High-severity item (a route-level error boundary that has been on the roadmap since the very first audit, six milestones ago, and still hasn't been picked up). Everything else found is Medium or lower, and the majority of areas checked are **Already acceptable** — confirming, not just asserting, that prior milestones' own claims of "done" hold up under independent re-verification.

Concretely: dead code (a real, recurring category of concern raised in F-4.4) is now **fully resolved** — the exhaustive sweep in this audit found zero orphaned exports anywhere in `components/`, `lib/store/`, or `lib/hooks/`, beyond the three already fixed in F-4.5. Duplicate abstractions are narrow and bounded (one real case, ~30-40 lines, not spreading). Theme-token consistency is 98%+ complete, and the two remaining exceptions are already explicitly documented in-code as deliberate, deferred decisions, not oversights. Keyboard infrastructure has zero drift from its F-3.1/F-3.2 design — no ad-hoc keyboard handling has crept into any of the workspaces that were deliberately left out of that rollout.

## 3. Dead Code Inventory

**None found, beyond what F-4.5 already fixed.** An exhaustive cross-check of every exported symbol in `frontend/components/**/*.tsx`, `frontend/lib/store/*`, and `frontend/lib/hooks/*` against its import/JSX call sites found real usage for all of them:

- Every grid/detail/panel pair (Images, Tables, Headings, Footnotes, Lists, Callouts, Validation, Metadata, OCR, Reading Order, Page Labels, Corrections, Readiness) has a live import and JSX call in `DocumentWorkspace.tsx`'s `renderSpecialView()` switch.
- Card components (`HeadingCard`, `TableCard`, `ImageCard`) and detail panels (`CalloutDetailPanel`, `FootnoteDetailPanel`, `ListDetailPanel`, `ImageDetailPanel`, etc.) all have call sites in both their own Grid and `ContextInspectorRail.tsx`/`SemanticNavTree.tsx`.
- `lib/api.ts`'s response-wrapper interfaces (`ValidationResponse`, `ImagesResponse`, etc.) have no import sites *outside* `lib/api.ts` — but each is used as the internal `Promise<T>` return-type annotation for its matching `api.getX()` method in the same file. Not dead, just never re-exported — correctly so, since consumers only need the resolved value's shape, not the wrapper type name.
- Every context provider, hook, and key-function (`footnoteKey`/`listKey`/`calloutKey`, `useObjectInspectorContext`, `NO_FLASH_THEME_SCRIPT`) has confirmed real call sites.

**Verdict: Already acceptable.** The dead-code category that motivated Phase F-4.4/F-4.5 is now closed, not just patched at the three sites originally found.

## 5. Integration Issues

- **Uneven keyboard coverage remains, exactly as F-3.1 disclosed and recommended** — `ReviewerWorkspace` has full list-review shortcuts; `ValidationIssueTable`, `ImageGrid`, `TableGrid`, `HeadingGrid`, and the rest have none, because none of them yet has a "current item" / roving-focus concept for the hook to drive. Re-confirmed via grep: zero keyboard-handling code exists in any of those files today. This is a known, deliberately-scoped incomplete rollout, not a new regression — classified **Medium** (tracked, intentional, matches its own prior honest disclosure).
- **Accessibility test coverage has not grown alongside new surface area.** Still exactly 6 of the ~39 originally-audited frontend areas have `jest-axe` coverage (unchanged since F-2.1), while F-4.5 added three entirely new special-view workspaces (Footnotes/Lists/Callouts) with zero new test coverage of their own — the existing `document-workspace.a11y.test.tsx` renders the workspace in its default/no-special-view state, so it does not exercise the new `renderSpecialView()` branches at all. Classified **Medium** — the components themselves reuse already-tested patterns (`ObjectInspectorFrame`, list+detail shape), so risk is low, but the coverage-to-surface-area ratio is quietly shrinking as the app grows.
- **Everything else integrates cleanly**: the sync contexts (`Selection`/`PdfViewport`/`MarkdownViewport`) are consumed consistently everywhere they're needed, the tab-pattern hook is applied consistently, the theme-token system is applied consistently (98%+, both exceptions documented), and the newly-wired Footnotes/Lists/Callouts special views follow the Images/Tables/Headings pattern exactly, with no bespoke deviation found.

## 6. Cleanup Recommendations

1. **Add a route-level `error.tsx`** (High priority per §2, but small — a single new file following the standard Next.js App Router convention). This is the one item genuinely worth doing before or alongside the next feature milestone, not because it's urgent in a crisis sense, but because it has now been deferred across six milestones despite being cheap, and every additional milestone adds more render surface that could throw with no recovery path.
2. **Optional, low-priority**: extract the duplicated status-tab-bar markup/state from `ReviewerWorkspace.tsx`/`CorrectionsPanel.tsx` into one shared component once (if ever) a third consumer appears — premature to extract for just two call sites per this project's own YAGNI/DRY balance (`coding-style.md`: "introduce abstractions when repetition is real, not speculative" — two sites with only ~30 lines each is a borderline case, not a clear one).
3. **Optional, very low priority**: a shared `emptyMessage` default for `CorrectionHistoryList` (removing the 6 duplicated literal strings) — cosmetic only, no functional benefit, purely a future-wording-change convenience.
4. **Not recommended as cleanup**: the Badge.tsx / DocxPreview.tsx hardcoded-color exceptions — both are already explicitly documented in `globals.css` as deliberate, incremental-migration decisions with clear rationale (Badge's tone colors; DocxPreview's theme-independent Word-preview surface). Re-litigating an already-made, well-reasoned decision would be busywork, not cleanup.

**None of the above rises to the size or urgency of a dedicated milestone.** Item 1 is a single small file; items 2-3 are two-line-item nice-to-haves with no correctness impact, explicitly not urgent by this project's own DRY threshold.

## 7. Recommended Implementation Order for Remaining Frontend Work

1. **`error.tsx` route-level error boundary** — trivial, standalone, no design decision needed, closes the one genuinely-aged gap.
2. **Context-preserving table/image editing** (F-4.2/F-4.4's top recommendation) — highest remaining reviewer-productivity payoff, no backend dependency, no open design question left after F-4.4's audit.
3. **Large Document Mode** — the single biggest named UX gap (F-4.2), now that the workspace has stable persistence (F-4.3), full object-type coverage (F-4.5), and a confirmed-sound architecture (this audit) to build it on top of. Should be preceded by the performance benchmark named in §2, per this project's "benchmark before generalizing" principle.
4. **Inspector Compression** — needs a first design pass before implementation; can proceed in parallel with #3 since it touches a different surface (`ObjectInspectorFrame`/`ContextInspectorRail` width behavior, not the PDF/nav-tree virtualization Large Document Mode needs).
5. **Real-time sync** — blocked on backend scoping (SSE/WebSocket or optimistic-update design); the frontend dispatch plumbing to receive push updates already exists (F-4.2), so this is backend-led, not frontend-blocked.
6. **Everything explicitly out of scope or backend-blocked in prior audits** (Responsive layout, Settings, Export Center consolidation, History surface, Performance baseline as a standalone deliverable rather than a pre-Large-Document-Mode gate) — unchanged priority, no new urgency found by this audit.

## 8. Updated Completion Estimate

Against the original 39-item Frontend Completion Matrix (2026-07-13): the "real, addressable, no-backend-required" gaps that matrix identified are now substantially closed — Search unification was audited and confirmed unnecessary (F-4.1), Keyboard workflow got its shared infrastructure plus one full consumer with the rest honestly scoped and documented rather than shallow-rolled (F-3.1/F-3.2), and Lists/Callouts/Footnotes workspace parity is done (F-4.5). Accessibility, the matrix's own highest-priority item, has real (if narrow) automated coverage plus a completed manual pass (F-2.1/F-2.2) — a defensible, if not total, close.

**Genuinely still open, unchanged by this audit:** route-level error boundaries (small, now flagged again as overdue), Export Center consolidation, History surface (backend-blocked), Settings (needs a product decision), a performance baseline. **Newly identified by the F-4.x design-audit thread, not part of the original matrix:** Large Document Mode, Inspector Compression, context-preserving object editing, real-time sync.

**Estimate: the "Phase 1-3 frontend implementation wave" this ticket describes as concluding is functionally complete.** What remains is either explicitly out of scope by product decision (responsive layout, multi-user features), blocked on a decision outside this audit's authority (Settings, real-time sync backend design), or is the well-scoped "final major feature wave" the F-4.x audits already mapped in detail (Large Document Mode, Inspector Compression, context-preserving editing) — not unknown or underspecified work.

## 9. Recommendation

**Begin the next feature milestone (context-preserving editing or Large Document Mode) rather than run a dedicated cleanup milestone — with the one exception of picking up `error.tsx` first, since it's small enough to not deserve its own milestone but has waited too long to defer again.**

This audit's own instruction was to attempt to prove further cleanup unnecessary before recommending it, and the evidence supports that conclusion for nearly everything checked: dead code is fully resolved, duplicate abstractions are narrow and non-spreading, the theme-token system is 98%+ consistent with the remaining exceptions already deliberately documented, and keyboard/tab infrastructure shows zero drift from its designed scope across six subsequent milestones. A dedicated cleanup milestone would be manufacturing work against a codebase that is, by direct re-verification rather than assertion, already in good order. The single exception — the error boundary — is real, but its fix is small enough to fold into the start of whichever feature milestone comes next rather than justify blocking on it alone.

## Devil's Advocate — what would prove this audit wrong

- **Assumption:** that the Explore-agent sweep for dead code and duplication was exhaustive rather than merely thorough. It checked every exported symbol in `components/`, `lib/store/`, and `lib/hooks/` against call sites — a systematic method, not sampling — but it is still a single automated pass; a human code review focused specifically on hunting for dead code could still find something a mechanical import-site check misses (e.g., a component imported but never actually reachable through any real user flow, which is a subtler kind of "dead" than zero import sites).
- **Assumption:** that the accessibility test coverage gap (6/39 areas, unchanged since F-2.1) is acceptable to leave as Medium rather than escalate to High. If RAWRS's own stated purpose (an accessibility tool auditing its own accessibility) is taken as seriously as the F-1 audit itself argued, a reasonable reader could conclude this deserves the same "gate, not backlog item" framing F-1 gave the original accessibility gap — this audit chose not to re-escalate it because the *new* surface (Footnotes/Lists/Callouts) reuses already-tested patterns rather than introducing new interaction models, but that's a judgment call, not a certainty.
- **Assumption:** that no performance regression has been introduced by the cumulative F-2 through F-4.5 additions (three new special views, two new Contexts' worth of state, `autoSaveId` persistence, a scroll-into-view effect). No profiling was run this session to confirm this — the conclusion rests on code review (no obviously expensive new render paths) plus the unchanged, still-passing test/build suite, not a measured before/after.
- **What would change this recommendation:** if a quick performance smoke-test (not a full Lighthouse pass, just opening the workspace with a large synthetic document) revealed a real slowdown from the cumulative additions, that would argue for a narrow, targeted performance-focused stabilization pass before Large Document Mode — smaller than a full "cleanup milestone" but larger than the `error.tsx` fix this audit does recommend.

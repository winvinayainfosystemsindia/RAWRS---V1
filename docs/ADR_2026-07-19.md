Adjudicates `ARCHITECTURE_INVENTORY_2026-07-19.md` (C-*) against `ARCHITECTURE_REVIEW_2026-07-19.md` (A-*). Neither is ratified wholesale: **3 inventory proposals rejected, 2 review proposals rejected or amended, 3 decisions supersede both.**

Accepted ADRs should be appended to `DECISIONS_LOG.md` as superseding entries, per that file's append-only convention.

## Finding that reopened the review's central claim

`phase1_pipeline.py:464,492` — RAWRS emits **`.md` and `.docx` only**. No tagged PDF, no EPUB. PDF/UA appears in `ACCESSIBILITY_INTELLIGENCE_ENGINE_DESIGN.md:93` purely as rule-mapping vocabulary.

The adversarial review argued the tree was mandatory because "both compliance targets are trees." **False.** The output target is DOCX, whose content model is largely flat: paragraphs carrying outline levels, lists encoded as `numId`/`ilvl` properties, only tables genuinely nested. The review over-argued the tree by treating an input standard as an output contract. ADR-001 corrects this.

# ADR-001 · Canonical Document Model

**Decision: ACCEPT WITH MODIFICATIONS.** A **shallow tree with guaranteed total linearization** — not a flat sequence (C-1), not a full recursive structure tree (A-1).

Nest only where nesting is real: `List > Item > (content, nested List)`, `Table > Row > Cell`, `Figure > Caption`, `Section > *`. Everything else is a sibling. Every tree **must** linearize to a total order via `walk()`; no node may exist outside that order.

| | |
|---|---|
| Problem | 12 parallel flat lists cannot express nesting; `P2Block.list_number` proves the flat model already failed |
| Evidence | `document.py:75-121` flat lists · `phase2_document.py:86-96` union-by-optional-fields, nested lists unrepresentable · DOCX needs `ilvl` depth, which requires knowing depth |
| Alternatives | **(a)** Flat sequence (C-1) — rejected, cannot encode list depth or cell scope. **(b)** Full PDF/UA structure tree (A-1) — rejected, models structure the output format cannot carry. **(c)** Shallow tree — **chosen** |
| Trade-offs | Less expressive than a full tree; if EPUB/HTML export is added later, sections deepen. Accepted: the linearization guarantee means deepening is additive |
| Migration | Build tree alongside existing lists; lists become `walk()` views. Retire lists only after ADR-014's byte-diff gate passes |
| Benchmark | Neutral if byte-diff holds; enables per-node scoring afterwards |
| Accessibility | Enables correct `ilvl` nesting and table header scope — both currently unexpressible |
| Reviewer workflow | Navigator becomes a real tree instead of a flattened list |
| Enterprise | Neutral |
| Future AI | Tree + linearization is the standard input shape for document-understanding models |
| Maintainability | One traversal replaces 12 list-grouping helpers |
| **Risk** | **High** — widest blast radius in the project |
| **Rollback** | Lists remain populated in parallel through the entire migration; revert = stop calling `walk()` |
| **Success** | Byte-identical `.md` and `.docx` on all 10 benchmark PDFs; nested list depth correct in DOCX |

# ADR-002 · Object Identity

**Decision: SUPERSEDED.** Both prior proposals rejected. Identity is **assigned once and persisted**, never derived: a UUID minted at first detection, re-anchored on reprocessing by a **multi-selector matcher** (content quote + structural path + page position, with fallback precedence).

| | |
|---|---|
| Problem | IDs are positional — `f"fn-{idx}"` (`footnote_detector.py:266`), `f"table-p{page}-{index}"` (`table_extractor.py:316,537`), `f"p{page}-{number}"` (`markdown_builder.py:452`). Insert one object and every downstream ID shifts, detaching correction history |
| Evidence | Verified in source. `Document.corrections` and every reviewer decision key off these IDs |
| Alternatives | **(a)** Positional (today) — data corruption. **(b)** Content hash (A-4) — **rejected: re-OCR changes content, so the hash changes precisely when stability matters most**; A-4 admitted this and left it unsolved. **(c)** Assigned UUID + re-anchoring — **chosen** |
| Precedent | W3C Web Annotation selectors solve exactly this: `TextQuoteSelector` + `TextPositionSelector` with fallback ordering. Anchoring annotations to mutating documents is a solved problem; do not reinvent it |
| Trade-offs | Requires a matcher and a persisted ID store; matcher can mis-anchor. Mitigate: mis-anchor is detectable (confidence below threshold → orphan, surfaced to reviewer, never silently reattached) |
| Migration | Mint UUIDs at detection; keep legacy ID as a display alias. **Independent of ADR-001 — ships first** |
| Benchmark | **Precondition.** Labels cannot accumulate against moving identifiers |
| Accessibility | Indirect but decisive: without it, review decisions silently attach to wrong objects |
| Reviewer workflow | Correction history survives reprocessing — currently it does not |
| Enterprise | Precondition for audit trails |
| Future AI | Precondition for any learning |
| **Risk** | Medium — matcher quality |
| **Rollback** | Legacy positional IDs retained as alias; revert = read the alias |
| **Success** | Re-run a benchmark PDF after an edit; >95% of prior corrections re-anchor, 0 silent mis-anchors |

# ADR-003 · Artifact Classification

**Decision: ACCEPT** (A-2). Content is partitioned into **Real Content** and **Artifact** (`pagination` / `layout` / `background`). Running headers, running footers and page numbers are Pagination Artifacts — **classified and retained, never deleted**.

| | |
|---|---|
| Problem | D1: first occurrence of a running header still becomes a heading (`heading_detector.py:550-554`, stated in its own docstring) |
| Evidence | Single-pass detection cannot see repeats ahead. `P2Heading.is_running_header` was designed in Phase 2 and never propagated |
| Alternatives | **(a)** Filter/remove (the brief's verb) — rejected: destroys auditability and reviewer override. **(b)** Transient `DocumentProfile` (C-2) — rejected: single-consumer stats bag, accretes into a god-object. **(c)** Persisted partition — **chosen** |
| Trade-offs | Heavier than a throwaway profile; needs an override surface. That surface is the deliverable |
| Migration | New pipeline stage between `detect_structure` and `detect_headings`. Additive; detector takes optional classification and falls back to current behaviour. **Independent of ADR-001** |
| Accessibility | Directly fixes heading pollution — the most reviewer-visible defect in the product |
| **Risk** | Low |
| **Rollback** | Detector ignores the classification |
| **Success** | D1 and D2 regression tests pass; benchmark heading counts do not regress on all 10 PDFs |

# ADR-004 · Physical Zones · **ACCEPT**

Per-page geometric zones: `header`, `footer`, `body`, `margin`, `gutter`. Populated by ADR-003's pass. Independent of ADR-001.
**Risk** Low · **Rollback** field ignored · **Success** running headers correctly zoned on all benchmark PDFs.

# ADR-005 · Logical Divisions · **ACCEPT**

`front_matter`, `body`, `appendix`, `back_matter` as **Section nodes in ADR-001's tree** — not an enum field.

**Rejects C-3's single `DocumentRegion` enum as a category error.** Physical zones and logical divisions are orthogonal: front-matter pages carry running headers. One enum forces a false choice per object, and cannot represent a sidebar (a physical zone *containing* logical content). This is why ALTO/PAGE-XML (physical) and TEI/JATS (logical) are separate standards.
**Depends on** ADR-001 · **Risk** Low · **Success** front matter correctly bounded on the benchmark corpus.

# ADR-006 · Relationships · **ACCEPT**

Closed, typed, non-tree edges only: `note_ref→note_body`, `xref→target`, `cell→header`, `continuation`, `figure→described_by`.

**Rejects C-4's open-vocabulary graph.** Most "relationships" are containment, which ADR-001 provides natively. The remainder already have standard vocabulary (PDF/UA `/Ref`, table `Headers`/`IDs`, `aria-describedby`). An open `kind` field invents a private ontology and rots — nobody can enumerate legal values.
**Depends on** ADR-001, ADR-002 · **Risk** Low · **Success** every edge type maps 1:1 to a DOCX construct.

# ADR-007 · Reading Order · **ACCEPT WITH MODIFICATIONS**

The tree **stores** reading order (in-order `walk()`). It does **not derive** it — multi-column and multi-block pages still need order *detection*, which stays a separate stage feeding tree construction.

The review's "reading order is free from traversal" was half-right: traversal is free, *correctness* is not. Existing `ReadingOrderStatus` + PAGE_003 + the review workspace are retained.
**Risk** Low · **Success** reading-order corrections persist as tree edits, not a parallel status field.

# ADR-008 · Rule Framework

**Decision: SUPERSEDED.** Both prior proposals rejected. **One registry, one output stream, `family` as a field** — not a merge of rule sets (C-6), not three separate registries (A-6).

| | |
|---|---|
| Problem | Two engines diverged: 35 hardcoded validator rules vs 13 registry rules, no shared vocabulary |
| Alternatives | **(a)** Merge rule sets (C-6) — rejected: destroys the distinction between an extraction defect and a remediation task. **(b)** Three registries (A-6) — rejected: reviewers see one queue; three registries is conservatism, and triples the wiring. **(c)** One registry + `family` tag — **chosen** |
| Families | `extraction` (did we read it right?) · `verification` (do sources agree?) · `accessibility` (is output accessible?) |
| Versioning | Every rule carries `rule_id@version`. **Mandatory** — without it a precision change cannot be attributed to rules vs documents |
| Migration | Port `validator.py`'s 35 rules to the registry interface, IDs unchanged so the frontend is untouched |
| **Risk** | Medium — mechanical but wide |
| **Rollback** | Per-rule; ported rules coexist with unported ones |
| **Success** | One registry, 48 rules, IDs stable, frontend unchanged |

# ADR-009 · Finding & Evidence Vocabulary · **ACCEPT**

`Finding` + `EvidenceBundle` (already converging in `verification/engine.py` and `accessibility/registry.py`) become the single output type for all three families. Carries `rule_id@version` and ADR-002 object ID.
**Risk** Low · **Success** one frontend issue type replaces the current parallel shapes.

# ADR-010 · Reviewer Decision Log · **ACCEPT (new)**

**Decision:** every reviewer decision is persisted as an **immutable append-only event**: `(object_id, rule_id@version, features_at_decision, decision, reviewer, timestamp)`.

**Design principle: capture now, model later.** The learning *model* is re-derivable at any time; the *events* are not. Every day without capture is permanently lost training data. This is why the log ships in 3.5B while the learning architecture stays deliberately undesigned.

| | |
|---|---|
| Problem | "Learn from expert remediators" has no data substrate. `Document.corrections` is mutable, in-memory, and keyed to positional IDs |
| Depends on | ADR-002 (stable identity), ADR-008 (rule versioning), ADR-012 (durability) |
| Alternatives | Design the learning model now — **rejected as premature**; the model cannot be chosen before the data exists to characterise it |
| **Risk** | Low |
| **Rollback** | Log is write-only and read by nothing initially; it cannot break the product |
| **Success** | Every accept/reject/edit produces exactly one durable event; replayable |

# ADR-011 · Evaluation Harness · **ACCEPT** (A-7)

Ground-truth corpus (`samples/benchmark/`, 10 PDFs) · replay detection · per-rule precision/recall · regression gate on merge.
**Rationale:** this project has repeatedly shipped features believed complete and later found broken. An eval harness is the structural fix for that pattern; more careful reports are not.
**Depends on** ADR-002, ADR-008 · **Risk** Low · **Success** a rule change reports its precision delta before merge.

# ADR-012 · Persistence · **ACCEPT WITH MODIFICATIONS**

`_jobs: Dict[str, Job] = {}` (`api/jobs.py:93`) is in-memory — **a restart loses every job**.

**Modification:** the review sequenced this as an enterprise blocker. Honest scoping: RAWRS is documented as **local-first, single-reviewer** (`RAWRS_PROJECT_CONTEXT.md`). For that deployment, in-memory is survivable. **But ADR-010's decision log must be durable regardless** — that data is unrecoverable.

**Decision:** durable store for decision log + corrections in 3.5B (SQLite; local-first, no new service). Full job persistence deferred to the phase preceding any multi-user deployment.
**Risk** Low · **Rollback** in-memory path retained · **Success** decision events survive restart.

# ADR-013 · Concurrency · **ACCEPT WITH MODIFICATIONS**

`Document.version` is documented as explicitly *not* a concurrency token (`document.py:113-120`). Two reviewers on one document clobber silently.

**Decision:** deferred, with a **hard gate** — no multi-reviewer deployment until optimistic concurrency exists. Documented as a known limitation, not a silent one. Single-reviewer local-first use is unaffected.
**Risk** Low now, **High if deployment model changes** · **Success** `KNOWN_LIMITATIONS.md` states the gate.

# ADR-014 · Dual Rendering Path · **ACCEPT**

Retire `_render_page_body_line_by_line()` (`markdown_builder.py:709`) in favour of object projection, once ADR-001 lands.

The native path rejoins objects to text by **exact string match**; the Mathpix path projects real objects (`:798-842`). The Mathpix docstring states the native mechanism "structurally cannot work" for object projection (`:809-812`). This is the central defect both prior documents agreed on — the only unanimous finding.
**Depends on** ADR-001 · **Risk** High · **Rollback** native path retained as fallback until gate passes · **Success** ADR-001's byte-diff gate.

# Final Answer

> **"Would you personally approve this architecture for enterprise development over the next several years?"**

**Yes — conditionally, and the conditions are the point.**

Approve because: the model matches the output contract rather than an aspirational one (ADR-001 corrected for DOCX reality); identity uses a solved anchoring pattern instead of a hash that breaks on the exact event it must survive (ADR-002); the artifact partition retains and explains rather than deletes, which is what remediation audit requires (ADR-003); the rule vocabulary is closed and versioned; and the decision log captures irreplaceable data before anyone knows what model will consume it.

**Conditions, in order:**
1. **ADR-013 is a hard gate.** This architecture is approved for *local-first, single-reviewer* use. It is **not** approved for multi-tenant deployment until optimistic concurrency exists. Deploying multi-user before then silently corrupts review data.
2. **ADR-001 lands behind its byte-diff gate or not at all.** The `SemanticObject` migration is 5-of-9 done and abandoned; a second stalled migration would leave three document models instead of two.
3. **ADR-011 before scale.** Without an eval harness, "semantic maturity" stays an opinion — and this project's record shows opinion has been wrong repeatedly.

**Honest reservation:** this is the third architectural document in one session, and each has overturned part of its predecessor — including this one overturning the review's central tree argument on a fact I could have checked in the first pass. Treat ADR-001 as the decision most likely to need revisiting, and revisit it the moment an EPUB or tagged-PDF output target appears, which would restore the full-tree argument I just rejected.

# RAWRS Document Workspace Layout Audit — 2026-07-14 (Phase F-4.2)

Audit + design only, per the ticket — no code changed, no visual redesign, no responsive layout implemented. Grounded in direct inspection of every panel under `frontend/components/workspace/`, `frontend/app/documents/[id]/DocumentWorkspace.tsx`, and every component it composes, cross-checked against `docs/FRONTEND_COMPLETION_AUDIT_2026-07-13.md` (Phase F-1), `docs/SEARCH_ARCHITECTURE_AUDIT_2026-07-13.md` (Phase F-4.1), and the accessibility/keyboard work landed since (F-2.1/F-2.2/F-3.1/F-3.2, commits through `55be14e`). ## 2. Gap Analysis

Ranked by how directly each gap contradicts the ticket's **named, user-approved UX requirements** (large-document mode, intelligent inspector compression, adequate preview space, real-time sync, hours-long-session efficiency, no dashboard clutter):

1. **Large Document Mode does not exist.** Zero virtualization, zero windowing, zero large-doc-specific navigation anywhere. This is the single most explicit named requirement in the ticket and the single biggest gap found.
2. **"Real-time" synchronization is actually 3–4 second polling.** Contradicts the ticket's named requirement directly; the architecture (canonical Document Model + on-demand render) is correct, but the frontend's visibility into it lags by design.
3. **Table/Image/Reading-Order/Page-Label editing silently drops PDF/Markdown context** when entered via the dedicated special-view grids (but not when entered via a PDF-overlay click or nav-tree row) — an unintentional inconsistency, not a designed tradeoff, and it hits exactly the "hours-long session" workflows the ticket names.
4. **Inspector has no compression logic at all** — the ticket asks for "compresses intelligently instead of hiding important controls"; today it neither compresses nor hides, it just CSS-wraps, which is a different (and untested-at-narrow-widths) behavior than either stated option.
5. **Zero layout persistence.** Every reload discards the reviewer's chosen split/focus-mode/panel-sizes — directly undercuts "efficient workflows for remediators working for hours," since "hours" implies multiple sessions across a workday, not one unbroken sitting.
6. **No `maxSize` on any resizable panel** — a lower-severity but trivially-fixed correctness gap (a panel can currently be dragged to near-zero width for its siblings).
7. **No scroll-to-highlight in the PDF pane** — jump targets can land off-screen on tall pages, a real but narrower-impact gap than the above.

**Not gaps — verified working:** Split View, Focus Mode (mechanically, minus persistence), Evidence Panel, History Panel (best-in-class reuse), Corrections Panel, workspace synchronization contexts (mature, loop-safe, well-commented), Markdown→Canonical→DOCX architecture itself (only its latency is a gap, not its design).

## 3. Existing Infrastructure Already Satisfying Requirements

- **`react-resizable-panels`** — already the shell's panel primitive; `autoSaveId` and `maxSize` (both unused) are already available in the installed version with zero new dependencies.
- **`useArrowKeyTabs`** (Phase F-3.2) — a single, accessible tab-switching hook already used by `WorkspaceShell`'s center-mode tabs, `SemanticNavTree`'s mode tabs, and `ObjectInspectorFrame`'s inspector tabs. Any inspector-compression redesign (icon-only tabs, overflow menu) can reuse this hook unchanged — it only needs `ids`/`active`/`onChange`, not a specific label-rendering shape.
- **`SelectionContext`/`PdfViewportContext`/`MarkdownViewportContext`** — mature cross-panel sync primitives, already engineered against the specific infinite-loop failure mode (reference-equality bailouts, `nonce`-based re-trigger for same-target jumps). A large-document mode or a context-preserving table/image editor can build on these directly rather than inventing new sync machinery.
- **`CorrectionHistoryList`** — the workspace's best existing example of "one component, several call sites, never diverging" — the template to follow when unifying the table/image context-loss inconsistency (item 3 above), rather than building two separate fixes for Tables and Images.
- **`MarkdownEditor`'s `scrollIntoView` pattern** — already solves exactly the problem `PdfViewer` doesn't (item 7 above); porting it is copying a proven pattern, not designing a new one.
- **`ThemeProvider`'s `localStorage` usage** — the only existing precedent for client-side persistence in this app; the same pattern (a few typed keys, read on mount, written on change) is the natural template for panel-layout persistence.

---

## 4. Missing Capabilities (Net-New, No Existing Precedent)

- Large Document Mode (virtualization/windowing/large-doc navigation) — nothing in the codebase does this today, for the PDF pane or the nav-tree lists.
- Push-based (SSE/WebSocket) document-version updates, or an optimistic local-update path — the poll interval is a deliberate `ponytail:`-flagged placeholder, not an oversight, but closing it requires new backend surface.
- A designed (not incidental) narrow-inspector state — icon-only tabs, an overflow menu, or an explicitly chosen minimum useful width, whichever direction is picked.
- A context-preserving editing mode for tables/images/reading-order/page-labels (i.e., a way to edit these without leaving the PDF/Markdown view entirely) — currently only available via the rail path, not the dedicated grid/special-view path.

---

## 5. Ranked Implementation Order

1. **Panel persistence (`autoSaveId` + a few `localStorage` keys for `centerMode`/`focusMode`/`bottomOpen`).** Trivial effort, directly serves the "hours-long session" requirement, zero backend dependency, zero design decision needed.
2. **`maxSize` on every resizable panel.** Trivial, prevents a real (if minor) layout-breaking interaction.
3. **Scroll-to-highlight in `PdfViewer`.** Small effort, ports an existing proven pattern (`MarkdownEditor`), closes a concrete "jump to X, then hunt for it" annoyance.
4. **Context-preserving table/image editing.** Medium effort, highest reviewer-productivity payoff of the remaining items — this is the one place a design decision is needed first: either (a) let the special-view grids render alongside a collapsed PDF strip, or (b) explicitly reframe the grid view as "bulk triage" and the rail view as "inspect with context," and message that distinction in the UI rather than leaving it accidental. Reading-order and page-label editing should follow whichever pattern is chosen here, since they share the identical structural issue.
5. **Inspector compression design + implementation.** Medium effort; needs a first design pass (this ticket's non-goal excludes actually building it, but the recommendation is to scope it next), then a mechanical implementation reusing `useArrowKeyTabs`.
6. **Large Document Mode.** Medium–large effort, the single biggest named gap; should follow the above because a virtualized nav-tree and a context-preserving inspector are easier to build on top of a workspace that already has stable, persisted layout and a designed narrow-inspector state, rather than layering all of this at once.
7. **Real-time sync (backend push or optimistic updates).** Requires backend work; sequence after the above because it's the most expensive item and the frontend dispatch plumbing to receive push updates already exists — this is the one item where backend scoping, not frontend design, is the bottleneck.

## 6. Accessibility Considerations

- Every gap above that involves a tab bar, overflow menu, or new interactive control must reuse `useArrowKeyTabs` (already WCAG-APG-compliant for the "tabs" pattern) rather than inventing new keyboard handling — consistent with Phase F-3.2's own precedent.
- Large Document Mode's navigation (item 6 in the ranking) has real accessibility upside, not just performance upside: a flat, uncapped list is already a worse screen-reader experience on a long document today, independent of any rendering-performance concern — virtualization done right (with correct `aria-setsize`/`aria-posinset` or an equivalent live-region announcement) would improve this, done wrong (virtualizing without accessible list semantics) would make it worse. This needs explicit attention when scoped, not an assumption that "faster" automatically means "as accessible."
- The context-preserving editing gap (item 4) has a real accessibility angle too: a keyboard-only reviewer using the rail path already gets full document context; a screen-reader user forced into the grid-only path for bulk alt-text review loses the PDF's visual layout as an orientation aid, which sighted reviewers on that same path also lose — this is a case where fixing productivity and fixing accessibility are the same fix, not competing priorities.
- No `ResizeObserver`-based compression should ship without confirming it doesn't break `useArrowKeyTabs`'s roving-tabindex focus management when the tab set re-renders at a new width — this needs a manual test pass when item 5 is implemented, not just a visual check.

## 7. Performance Considerations

- Large Document Mode is the one item on this list that is a performance problem as much as a UX one: `SemanticNavTree`'s flat, uncapped `by-type` lists and `PdfViewer`'s per-object overlay rendering (one absolutely-positioned `<button>` per semantic object, per page) have not been measured against a genuinely large document — no benchmark exists today (matches the 2026-07-13 completion audit's item 39 finding, which is project-wide, not workspace-specific). Recommend establishing a large-document benchmark corpus entry *before* committing to a specific virtualization approach, per this project's own "benchmark before generalizing" principle.
- `DocxPreview`'s full re-fetch-and-reconvert on every `document_version` bump (no caching, no `AbortController`) is a real but low-impact cost today; it becomes a more relevant cost once real-time sync (item 7) shortens the interval between version bumps, since more frequent bumps would mean more frequent full DOCX re-conversions. Worth revisiting together with the real-time sync work, not in isolation.
- None of the identified gaps require a new rendering library, state-management approach, or architectural layer — every fix listed reuses primitives already installed and already proven in this codebase.

---

## 8. Recommendation: Incremental or Architectural Redesign?

**Incremental improvements are sufficient. No architectural redesign is warranted.**

The panel shell (`react-resizable-panels`), the cross-panel sync layer (`SelectionContext`/`PdfViewportContext`/`MarkdownViewportContext`), and the shared-component reuse patterns (`ObjectInspectorFrame`, `CorrectionHistoryList`, `useArrowKeyTabs`) are all sound, already battle-tested against real bugs (the M-4.2 infinite-loop fixes, found via live browser verification, are direct evidence the sync architecture has already been stress-tested), and directly reusable for every gap identified in this audit — including the two most consequential ones (Large Document Mode and context-preserving editing). Nothing found here requires replacing the workspace shell, the sync contexts, or the inspector/nav-tree component structure; every recommended fix is additive (a new prop, a ported pattern, a new mode built on existing primitives) rather than a rewrite of existing panels.

The two items serious enough to have been mistaken for architectural problems — Large Document Mode and "real-time" sync — are both, on inspection, **scoping gaps, not design flaws**: Large Document Mode was simply never built (there is no wrong virtualization approach in the code today to fix, only an absent one), and the sync latency is an explicit, commented, deliberate placeholder (`ponytail:` polling comment naming its own upgrade path) rather than an accidental architectural limitation.

## Devil's Advocate — what would prove this audit wrong

- **Assumption:** that the table/image grid-vs-rail context-loss inconsistency is unintentional. If product intent was always "grid view = bulk triage without document reference, rail view = point inspection with reference" — a deliberate two-mode design — then item 3 (Gap Analysis) is not a bug to fix but a distinction to document and message clearly instead. This audit could not confirm which is true from code alone; it should be confirmed with whoever made the original IDE-redesign UX call (Phase 1 IDE redesign) before treating it as a defect.
- **Assumption:** that 3–4 second poll latency is actually felt as a problem by reviewers. No usage telemetry exists (same blind spot the F-1 Frontend Completion Audit already named for its own Keyboard Workflow finding) — if reviewers rarely check the DOCX/Markdown pane immediately after an edit, this gap is real but lower-urgency than ranked here.
- **Assumption:** that Large Document Mode should be built as list virtualization + PDF navigation aids. An equally valid alternative reading of "expandable large document mode" is a per-object pagination/batch-loading API change on the backend (only load headings/tables/images for the currently-visible page range) rather than a frontend-only virtualization fix — this would change the backend-dependency answer for that row from "No" to "Yes" and should be settled before implementation begins, not assumed by this audit.
- **What would change this recommendation:** if a genuinely huge test document (multi-thousand-page or multi-thousand-object) were benchmarked and revealed the current architecture (context re-renders, full-list mapping, per-page single-view PDF rendering) breaks down in a way that isn't a simple "add windowing" fix — e.g., if the Context providers themselves became a re-render bottleneck at scale — that would be evidence for a real architectural change, not merely a missing feature. No such benchmark exists yet; this is exactly why the Performance Considerations section above recommends building one before committing to an implementation approach.
